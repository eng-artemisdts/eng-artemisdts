O conhecimento por trás dos padrões

Antes dos desafios, a base. Esta parte explica o que são padrões de projeto, percorre os cinco princípios SOLID e detalha cada um dos 23 padrões da Gang of Four com intenção, problema, solução em TypeScript e — o mais importante — quando não usá-los.

O que é um padrão de projeto?

Um padrão de projeto (design pattern) é uma solução comprovada e reutilizável para um problema recorrente de design de software. Ele não é um trecho de código pronto para colar: é um template, uma receita de estrutura e colaboração entre objetos que você adapta ao seu contexto.

O termo foi popularizado em 1994 pelo livro Design Patterns: Elements of Reusable Object-Oriented Software, escrito por Erich Gamma, Richard Helm, Ralph Johnson e John Vlissides — os quatro autores que ficaram conhecidos como a Gang of Four (GoF). A ideia foi inspirada no trabalho do arquiteto Christopher Alexander, que em A Pattern Language (1977) descreveu padrões recorrentes na arquitetura de prédios e cidades. Três décadas depois, os 23 padrões do catálogo continuam relevantes.

O maior valor de um padrão talvez seja o vocabulário compartilhado. Quando você diz a um colega "aqui cabe um Strategy" ou "isso é um Observer", uma estrutura inteira de intenção é comunicada em duas palavras.

Por que vale a pena estudar?

  • Vocabulário comum — comunicação técnica precisa entre o time e em revisões de código.
  • Soluções testadas — você evita reinventar a roda e cair em armadilhas já mapeadas.
  • Manutenção e extensão — código desacoplado é mais fácil de mudar sem quebrar o resto.
  • Você passa a reconhecê-los na prática — o middleware do Express é um Chain of Responsibility; o RxJS é Observer; os providers do NestJS são injeção de dependência (DIP); streams do Node combinam Decorator e Iterator.
Padrão é meio, não fim

Padrões resolvem problemas reais — aplicá-los sem necessidade gera over-engineering: camadas de abstração que ninguém pediu e que tornam o código mais difícil de ler. Em TypeScript, vários padrões clássicos de OOP são substituídos por recursos da própria linguagem (funções de primeira classe, union types, objetos literais). Fechamos a Parte I exatamente com esse tema, em "Você talvez não precise".

As três famílias da Gang of Four

Os 23 padrões são organizados em três categorias, segundo o tipo de problema que resolvem:

  • Criacionais (5) — lidam com a criação de objetos, separando o "como criar" do "como usar".
  • Estruturais (7) — lidam com a composição de classes e objetos em estruturas maiores.
  • Comportamentais (11) — lidam com a comunicação e a divisão de responsabilidades entre objetos.

Os cinco princípios SOLID

Antes dos padrões, os princípios. SOLID é um acrônimo (cunhado por Michael Feathers a partir dos artigos de Robert C. Martin, por volta de 2000) para cinco diretrizes de design orientado a objetos que tornam o código mais legível, flexível e fácil de manter. Eles são a fundação sobre a qual quase todos os padrões da GoF se apoiam — e o vocabulário que justifica por que um padrão melhora o design.

S
Princípio SOLID

Responsabilidade Única

Single Responsibility Principle (SRP)
Uma classe deve ter uma — e apenas uma — razão para mudar.

Cada módulo deve ser responsável por uma única parte da funcionalidade. Quando uma classe acumula responsabilidades (calcular, persistir e notificar, por exemplo), uma mudança em qualquer uma delas força a abrir e arriscar quebrar as outras. Separar responsabilidades gera componentes coesos, fáceis de testar e de evoluir isoladamente.

Antes → Depois (TypeScript)
srp.ts · typescript
// ❌ Três razões para mudar dentro de uma classe só
class Invoice {
  constructor(public items: { price: number; qty: number }[]) {}
  total(): number {
    return this.items.reduce((s, i) => s + i.price * i.qty, 0);
  }
  saveToDatabase(): void { /* SQL, conexão, transação... */ }
  sendByEmail(to: string): void { /* SMTP, template... */ }
}

// ✅ Uma responsabilidade por classe
class Invoice {
  constructor(public items: { price: number; qty: number }[]) {}
  total(): number {
    return this.items.reduce((s, i) => s + i.price * i.qty, 0);
  }
}
class InvoiceRepository {
  save(invoice: Invoice): Promise<void> { /* ...persistência... */ }
}
class InvoiceMailer {
  send(invoice: Invoice, to: string): Promise<void> { /* ...e-mail... */ }
}

Na prática: se você descreve o que a classe faz e precisa usar a palavra "e" ("calcula o total e envia o e-mail"), provavelmente há mais de uma responsabilidade ali.

O
Princípio SOLID

Aberto/Fechado

Open/Closed Principle (OCP)
Entidades de software devem ser abertas para extensão, mas fechadas para modificação.

Você deve conseguir adicionar comportamento novo sem editar código que já funciona e já foi testado. O mecanismo costuma ser uma abstração (interface ou classe abstrata) que novas implementações estendem. O sinal clássico de violação é um switch/if-else que cresce a cada novo caso.

Antes → Depois (TypeScript)
ocp.ts · typescript
// ❌ Cada novo tipo de cliente exige editar o método
class DiscountCalculator {
  calculate(customer: { type: string }, amount: number): number {
    switch (customer.type) {
      case "regular": return amount;
      case "premium": return amount * 0.9;
      // novo caso => modificar esta classe (fechado à extensão)
      default:        return amount;
    }
  }
}

// ✅ Aberto à extensão via abstração, fechado à modificação
interface DiscountPolicy { apply(amount: number): number; }
class NoDiscount      implements DiscountPolicy { apply(a: number) { return a; } }
class PremiumDiscount implements DiscountPolicy { apply(a: number) { return a * 0.9; } }
class SeasonalDiscount implements DiscountPolicy { apply(a: number) { return a * 0.8; } }

class DiscountCalculator {
  calculate(policy: DiscountPolicy, amount: number): number {
    return policy.apply(amount); // novas políticas não tocam aqui
  }
}

Na prática: OCP e o padrão Strategy são quase a mesma ideia vista de ângulos diferentes — guarde essa conexão para o Dia 18.

L
Princípio SOLID

Substituição de Liskov

Liskov Substitution Principle (LSP)
Objetos de um subtipo devem poder substituir os do tipo base sem quebrar o programa.

Formulado por Barbara Liskov em 1987, este é o princípio mais sutil. Se B estende A, qualquer código que espera um A deve funcionar ao receber um B — sem surpresas, exceções inesperadas ou pré/pós-condições mais rígidas. Herança que precisa lançar "não suportado" ou ignorar parâmetros é um forte sinal de violação.

Antes → Depois (TypeScript)
lsp.ts · typescript
// ❌ Penguin "é um" Bird, mas não voa — quebra quem chama bird.fly()
class Bird { fly(): void { /* ... */ } }
class Penguin extends Bird {
  fly(): never { throw new Error("Pinguins não voam"); }
}
function migrate(b: Bird) { b.fly(); } // estoura com Penguin

// ✅ Modele apenas o que é de fato substituível
interface Bird { eat(): void; }
interface FlyingBird extends Bird { fly(): void; }

class Sparrow implements FlyingBird { eat() {} fly() {} }
class Penguin implements Bird     { eat() {} swim() {} }

function migrate(b: FlyingBird) { b.fly(); } // só aceita quem voa

Na prática: "é um" da linguagem natural nem sempre vira herança. Pergunte se o subtipo cumpre todo o contrato — comportamento, não só estrutura.

I
Princípio SOLID

Segregação de Interface

Interface Segregation Principle (ISP)
Nenhum cliente deve ser forçado a depender de métodos que não usa.

Prefira muitas interfaces pequenas e específicas a uma única interface "gorda". Quando uma classe implementa um contrato grande, ela carrega métodos irrelevantes — que viram implementações vazias ou que lançam erro, poluindo o design e enganando quem lê o tipo.

Antes → Depois (TypeScript)
isp.ts · typescript
// ❌ Interface gorda força métodos não suportados
interface Machine {
  print(doc: string): void;
  scan(doc: string): void;
  fax(doc: string): void;
}
class OldPrinter implements Machine {
  print(doc: string) { /* ok */ }
  scan(_: string) { throw new Error("não suportado"); }
  fax(_: string)  { throw new Error("não suportado"); }
}

// ✅ Interfaces pequenas, combináveis sob demanda
interface Printer { print(doc: string): void; }
interface Scanner { scan(doc: string): void; }
interface Fax     { fax(doc: string): void; }

class SimplePrinter implements Printer { print(doc: string) {} }
class AllInOne implements Printer, Scanner, Fax {
  print(doc: string) {} scan(doc: string) {} fax(doc: string) {}
}

Na prática: o sistema de tipos do TypeScript adora interfaces pequenas — elas compõem com A & B e mantêm cada classe honesta sobre o que realmente faz.

D
Princípio SOLID

Inversão de Dependência

Dependency Inversion Principle (DIP)
Módulos de alto nível não devem depender de módulos de baixo nível; ambos devem depender de abstrações.

E mais: abstrações não devem depender de detalhes — detalhes é que dependem de abstrações. Na prática, sua regra de negócio (alto nível) não deveria instanciar diretamente um banco, um cliente HTTP ou um serviço de e-mail (baixo nível). Ela depende de uma interface, e a implementação concreta é injetada. É o coração da injeção de dependência e de arquiteturas como a Hexagonal e a Clean.

Antes → Depois (TypeScript)
dip.ts · typescript
// ❌ Alto nível acoplado a um detalhe concreto
class MySQLUserRepository {
  findById(id: string): Promise<User | null> { /* ... */ }
}
class UserService {
  private repo = new MySQLUserRepository(); // acoplamento rígido
  getUser(id: string) { return this.repo.findById(id); }
}

// ✅ Ambos dependem da abstração; o detalhe entra por construtor
interface UserRepository {
  findById(id: string): Promise<User | null>;
}
class UserService {
  constructor(private readonly repo: UserRepository) {}
  getUser(id: string) { return this.repo.findById(id); }
}

// troque a implementação sem tocar na regra de negócio
const service = new UserService(new MySQLUserRepository());
// ...e em testes: new UserService(new InMemoryUserRepository());

Na prática: é por isso que esse princípio aparece em todo o seu trabalho com NestJS — providers registrados por interface são DIP institucionalizado pelo framework.

Padrões Criacionais

Os cinco padrões criacionais tratam de como objetos vêm ao mundo. Eles desacoplam o código que usa um objeto do código que o cria, dando flexibilidade para trocar implementações, montar objetos complexos com segurança e controlar a quantidade de instâncias.

01
Padrão Criacional

Factory Method

Factory Method
Define uma interface para criar um objeto, mas deixa as subclasses decidirem qual classe instanciar.

Em vez de chamar new espalhado pelo código, você concentra a criação em um método que pode ser sobrescrito. O código cliente trabalha com a abstração do produto; a subclasse escolhe o produto concreto.

Solução (TypeScript)
factory-method.ts · typescript
interface Button { render(): string; }
class HtmlButton    implements Button { render() { return "<button>OK</button>"; } }
class WindowsButton implements Button { render() { return "[ OK ]"; } }

// O criador declara o factory method; subclasses fornecem o produto
abstract class Dialog {
  protected abstract createButton(): Button;     // <- factory method
  render(): string {
    const button = this.createButton();          // não sabe a classe concreta
    return `Dialog -> ${button.render()}`;
  }
}
class WebDialog     extends Dialog { protected createButton() { return new HtmlButton(); } }
class DesktopDialog extends Dialog { protected createButton() { return new WindowsButton(); } }

const dialog: Dialog =
  process.platform === "win32" ? new DesktopDialog() : new WebDialog();
console.log(dialog.render());
◤ Use quando
  • Você não sabe de antemão o tipo exato a criar (depende de configuração, plataforma, input).
  • Quer dar a subclasses um ponto de extensão para o produto.
  • Deseja isolar o código de criação para reaproveitar e testar.
◦ Evite quando
  • A criação é trivial e estável — um new direto é mais honesto.
  • Só existe uma variação de produto e nenhuma perspectiva de mais.
02
Padrão Criacional

Abstract Factory

Abstract Factory
Cria famílias de objetos relacionados sem especificar suas classes concretas.

É o "Factory Method elevado a famílias". Quando vários produtos precisam ser coerentes entre si (todos do tema claro, ou todos do tema escuro), uma fábrica abstrata garante que você nunca misture um botão claro com um checkbox escuro.

Solução (TypeScript)
abstract-factory.ts · typescript
interface Button   { render(): string; }
interface Checkbox { render(): string; }

interface UIFactory {                 // fábrica abstrata
  createButton(): Button;
  createCheckbox(): Checkbox;
}

class LightButton   implements Button   { render() { return "botão claro"; } }
class LightCheckbox implements Checkbox { render() { return "checkbox claro"; } }
class DarkButton    implements Button   { render() { return "botão escuro"; } }
class DarkCheckbox  implements Checkbox { render() { return "checkbox escuro"; } }

class LightFactory implements UIFactory {
  createButton()   { return new LightButton(); }
  createCheckbox() { return new LightCheckbox(); }
}
class DarkFactory implements UIFactory {
  createButton()   { return new DarkButton(); }
  createCheckbox() { return new DarkCheckbox(); }
}

function buildUI(factory: UIFactory) {
  return [factory.createButton().render(), factory.createCheckbox().render()];
}
console.log(buildUI(new DarkFactory())); // sempre coerentes
◤ Use quando
  • O sistema precisa funcionar com várias famílias de produtos intercambiáveis (temas, plataformas, fornecedores).
  • É essencial garantir que produtos de uma mesma família sejam usados juntos.
◦ Evite quando
  • Existe só uma família — você está adicionando peso sem ganho.
  • As famílias mudam de forma muito frequente (cada produto novo exige tocar todas as fábricas).
03
Padrão Criacional

Builder

Builder
Constrói objetos complexos passo a passo, separando a construção da representação final.

Quando um objeto tem muitos parâmetros — vários opcionais — um construtor gigante vira um campo minado de argumentos posicionais. O Builder oferece uma API fluente e legível, e pode validar invariantes no build().

Solução (TypeScript)
builder.ts · typescript
interface HttpRequest {
  url: string;
  method: string;
  headers: Record<string, string>;
  body?: unknown;
}

class RequestBuilder {
  private req: HttpRequest = { url: "", method: "GET", headers: {} };

  url(u: string)            { this.req.url = u; return this; }
  method(m: string)         { this.req.method = m; return this; }
  header(k: string, v: string) { this.req.headers[k] = v; return this; }
  json(body: unknown) {
    this.req.body = body;
    this.req.headers["Content-Type"] = "application/json";
    return this;
  }
  build(): HttpRequest {
    if (!this.req.url) throw new Error("url é obrigatória");
    return this.req;
  }
}

const request = new RequestBuilder()
  .url("/api/tracks")
  .method("POST")
  .header("Authorization", "Bearer ...")
  .json({ title: "Clair de Lune" })
  .build();
◤ Use quando
  • A criação envolve muitas etapas ou muitos parâmetros opcionais.
  • Você quer uma API legível (.url().method().build()) e validação centralizada.
  • O mesmo processo de construção precisa gerar representações diferentes.
◦ Evite quando
  • O objeto tem poucos campos — um objeto literal ou construtor simples basta.
  • Você não precisa de validação nem de variações na montagem.
04
Padrão Criacional

Prototype

Prototype
Cria novos objetos clonando um protótipo existente, em vez de instanciar do zero.

Útil quando criar um objeto é caro (consultas, cálculos) ou quando você quer copiar um estado já configurado. O ponto de atenção em JavaScript é a diferença entre cópia rasa e profunda — campos mutáveis precisam ser realmente clonados.

Solução (TypeScript)
prototype.ts · typescript
interface Cloneable<T> { clone(): T; }

class Config implements Cloneable<Config> {
  constructor(
    public env: string,
    public flags: Record<string, boolean>,
  ) {}

  clone(): Config {
    // structuredClone faz cópia profunda dos campos mutáveis (Node 17+)
    return new Config(this.env, structuredClone(this.flags));
  }
}

const base = new Config("prod", { newCheckout: true });
const staging = base.clone();
staging.env = "staging";
staging.flags.newCheckout = false;   // não vaza para "base"

console.log(base.flags.newCheckout);    // true
console.log(staging.flags.newCheckout); // false
◤ Use quando
  • Instanciar do zero é caro e você já tem um objeto pronto para copiar.
  • Você quer evitar uma hierarquia de fábricas paralela às suas classes.
  • Precisa de muitos objetos quase idênticos com pequenas variações.
◦ Evite quando
  • A construção é barata e simples.
  • O objeto tem referências complexas/circulares que tornam a clonagem arriscada.
05
Padrão Criacional

Singleton

Singleton
Garante que uma classe tenha apenas uma instância e oferece um ponto de acesso global a ela.

Clássico para recursos compartilhados como pool de conexões, configuração ou logger. É também o padrão mais controverso: ele introduz estado global, acopla o código a um ponto fixo e dificulta testes (cada teste compartilha a mesma instância).

Solução (TypeScript)
singleton.ts · typescript
class Logger {
  private static instance: Logger;
  private constructor() {}                  // impede "new" externo
  static getInstance(): Logger {
    return (Logger.instance ??= new Logger());
  }
  log(msg: string) { console.log(`[log] ${msg}`); }
}

Logger.getInstance().log("app iniciada");

// Em Node, um módulo ES já é um singleton por natureza:
// exportar uma instância costuma ser mais simples — e mais testável.
//   export const logger = new Logger();
Pense duas vezes antes de usar

Singleton é frequentemente um anti-pattern disfarçado: ele esconde dependências (quem usa Logger.getInstance() não declara que depende dele) e torna o estado global. Em Node/TypeScript, prefira exportar uma única instância de um módulo e injetá-la (DIP) onde for necessária — você ganha o "uma instância" sem perder a testabilidade. Voltamos a isso no Dia 10.

◤ Use quando
  • Realmente deve existir um único ponto de coordenação (ex.: registro central, cache de processo).
  • O acesso global simplifica mais do que complica e o estado é imutável/idempotente.
◦ Evite quando
  • Você só quer "acesso fácil" — isso é conveniência, não necessidade de instância única.
  • O objeto guarda estado mutável que vários testes/contextos vão compartilhar.
  • Dá para injetar a dependência — quase sempre dá.

Padrões Estruturais

Os sete padrões estruturais tratam de como compor classes e objetos em estruturas maiores, mantendo essas estruturas flexíveis e eficientes. A pergunta que eles respondem é: como conectar peças que não foram feitas para se encaixar — ou como organizar muitas peças — sem criar um emaranhado rígido? A maioria depende fortemente de composição em vez de herança.

06
Padrão Estrutural

Adapter

Adapter
Converte a interface de uma classe em outra que o cliente espera, fazendo interfaces incompatíveis trabalharem juntas.

É o padrão do plugue universal: você tem um código que fala uma interface e uma biblioteca (ou sistema legado) que fala outra. Em vez de reescrever qualquer um dos lados, você cria um adaptador no meio. É um dos padrões mais comuns ao integrar SDKs de terceiros.

Solução (TypeScript)
adapter.ts · typescript
// Interface que a nossa aplicação espera
interface PaymentGateway {
  pay(amountInCents: number): Promise<{ id: string }>;
}

// SDK de terceiros, com uma API totalmente diferente (não podemos mudá-la)
class StripeSdk {
  async createCharge(input: { value: number; currency: string }) {
    return { charge_id: "ch_123", value: input.value };
  }
}

// Adapter: traduz a nossa interface para a do SDK
class StripeAdapter implements PaymentGateway {
  constructor(private sdk: StripeSdk) {}
  async pay(amountInCents: number) {
    const res = await this.sdk.createCharge({
      value: amountInCents,
      currency: "brl",
    });
    return { id: res.charge_id };          // normaliza o retorno
  }
}

// O resto do sistema só conhece PaymentGateway
async function checkout(gateway: PaymentGateway) {
  return gateway.pay(4990);
}
checkout(new StripeAdapter(new StripeSdk()));
◤ Use quando
  • Você precisa usar uma classe existente cuja interface não bate com a do seu código.
  • Quer isolar a aplicação de detalhes de uma biblioteca de terceiros (e poder trocá-la depois).
  • Está integrando sistemas legados sem poder alterá-los.
◦ Evite quando
  • Você controla os dois lados e pode simplesmente alinhar as interfaces na fonte.
  • A tradução é tão complexa que esconde bugs — às vezes refatorar é melhor que adaptar.

Bridge

Bridge
Desacopla uma abstração da sua implementação, de modo que as duas possam variar de forma independente.

O Bridge ataca a explosão combinatória da herança. Imagine Notificação × Canal: se você cria NotificacaoUrgentePorEmail, NotificacaoUrgentePorSMS, NotificacaoNormalPorEmail... o número de classes cresce com o produto das dimensões. O Bridge separa as duas dimensões em hierarquias distintas ligadas por composição.

Solução (TypeScript)
bridge.ts · typescript
// Dimensão 1 — implementação (o "como enviar")
interface Channel {
  send(message: string): void;
}
class EmailChannel implements Channel {
  send(m: string) { console.log(`[email] ${m}`); }
}
class SmsChannel implements Channel {
  send(m: string) { console.log(`[sms] ${m}`); }
}

// Dimensão 2 — abstração (o "o que enviar"), ligada por composição
abstract class Notification {
  constructor(protected channel: Channel) {}   // a "ponte"
  abstract notify(text: string): void;
}
class UrgentNotification extends Notification {
  notify(text: string) { this.channel.send(`URGENTE: ${text}`); }
}
class SilentNotification extends Notification {
  notify(text: string) { this.channel.send(text); }
}

// Combine livremente, sem criar uma classe por combinação
new UrgentNotification(new SmsChannel()).notify("servidor caiu");
new SilentNotification(new EmailChannel()).notify("backup concluído");
◤ Use quando
  • Você tem duas (ou mais) dimensões que variam de forma independente (abstração × plataforma, forma × cor).
  • Quer evitar uma hierarquia de herança que cresce multiplicativamente.
  • Precisa trocar a implementação em tempo de execução.
◦ Evite quando
  • Existe só uma dimensão de variação — a herança simples resolve.
  • As dimensões estão tão acopladas que separá-las só adiciona indireção.

Composite

Composite
Compõe objetos em estruturas de árvore e permite tratar objetos individuais e composições de forma uniforme.

Sempre que você tem uma estrutura parte-todo — arquivos e pastas, itens e caixas, um menu com submenus — o Composite deixa o cliente tratar uma folha e um galho pela mesma interface. A recursão fica encapsulada na própria estrutura.

Solução (TypeScript)
composite.ts · typescript
// Componente comum a folhas e compostos
interface FileSystemNode {
  readonly name: string;
  size(): number;        // mesma operação para arquivo e pasta
}

// Folha
class FileLeaf implements FileSystemNode {
  constructor(public name: string, private bytes: number) {}
  size() { return this.bytes; }
}

// Composto: contém outros nós (inclusive outros compostos)
class Directory implements FileSystemNode {
  private children: FileSystemNode[] = [];
  constructor(public name: string) {}
  add(node: FileSystemNode) { this.children.push(node); return this; }
  size() {
    return this.children.reduce((total, child) => total + child.size(), 0);
  }
}

const root = new Directory("src")
  .add(new FileLeaf("index.ts", 1200))
  .add(new Directory("utils").add(new FileLeaf("math.ts", 800)));

console.log(root.size()); // 2000 — percorre a árvore recursivamente
◤ Use quando
  • Seus dados formam naturalmente uma hierarquia/árvore (parte-todo).
  • Você quer que o cliente ignore a diferença entre item simples e composição.
  • As operações devem se propagar recursivamente pela estrutura.
◦ Evite quando
  • A estrutura é plana — uma lista simples basta.
  • Folhas e compostos têm comportamentos tão diferentes que uma interface comum vira ficção.

Decorator

Decorator
Anexa responsabilidades a um objeto dinamicamente, oferecendo uma alternativa flexível à herança para estender comportamento.

O Decorator envolve um objeto em outro que adiciona comportamento antes ou depois de delegar para o original — e como ambos implementam a mesma interface, dá para empilhar quantos decoradores quiser. É exatamente como os streams do Node funcionam (um stream de gzip envolvendo um stream de arquivo).

Solução (TypeScript)
decorator.ts · typescript
interface DataSource {
  read(): string;
}

// Componente concreto
class RawData implements DataSource {
  constructor(private payload: string) {}
  read() { return this.payload; }
}

// Decorator base: implementa a interface e guarda o envolvido
abstract class DataSourceDecorator implements DataSource {
  constructor(protected wrappee: DataSource) {}
  abstract read(): string;
}

class UpperCaseDecorator extends DataSourceDecorator {
  read() { return this.wrappee.read().toUpperCase(); }
}
class BracketsDecorator extends DataSourceDecorator {
  read() { return `[${this.wrappee.read()}]`; }
}

// Empilhe responsabilidades por composição, em runtime
const source = new BracketsDecorator(
  new UpperCaseDecorator(new RawData("clair de lune")),
);
console.log(source.read()); // [CLAIR DE LUNE]
Em TypeScript, às vezes basta uma função

Para adicionar comportamento a funções (e não a objetos com estado), composição de funções de ordem superior costuma ser mais simples: const logged = withLogging(withRetry(fetchUser)). Reserve o Decorator clássico para quando há estado e uma interface rica a preservar.

◤ Use quando
  • Você precisa adicionar/remover responsabilidades em tempo de execução, em qualquer combinação.
  • A herança geraria uma explosão de subclasses para cada combinação de recursos.
  • Quer manter a interface original intacta para o cliente.
◦ Evite quando
  • Há poucas combinações fixas e conhecidas — subclasses diretas são mais legíveis.
  • O comportamento a adicionar é uma transformação de função pura (prefira composição funcional).

Facade

Facade
Oferece uma interface unificada e simples para um conjunto de interfaces de um subsistema complexo.

A Facade é uma recepção: o cliente fala com um único objeto de alto nível, que orquestra várias classes internas complicadas. Ela não esconde o subsistema (quem precisar ainda pode usá-lo direto) — apenas oferece o caminho fácil para os casos comuns.

Solução (TypeScript)
facade.ts · typescript
// Subsistema complexo — várias peças de baixo nível
class VideoFile { constructor(public name: string) {} }
class CodecFactory { extract(f: VideoFile) { return "h264"; } }
class BitrateReader { read(f: VideoFile, codec: string) { return f.name + ":" + codec; } }
class AudioMixer { fix(data: string) { return data + "+audio"; } }

// Facade: um método simples que esconde a orquestração
class VideoConverter {
  convert(filename: string, format: string): string {
    const file = new VideoFile(filename);
    const codec = new CodecFactory().extract(file);
    const buffer = new BitrateReader().read(file, codec);
    const result = new AudioMixer().fix(buffer);
    return `${result} -> ${format}`;
  }
}

// O cliente não precisa conhecer codecs, bitrate nem áudio
const out = new VideoConverter().convert("aula.ogg", "mp4");
◤ Use quando
  • Você quer um ponto de entrada simples para um subsistema complicado.
  • Precisa desacoplar o cliente de muitas classes internas (reduzir a superfície de dependência).
  • Quer organizar um sistema em camadas, com a facade como fronteira de cada camada.
◦ Evite quando
  • O subsistema já é simples — a facade só adiciona uma camada inútil.
  • Os clientes precisam de controle fino sobre as peças internas (a facade esconderia demais).

Flyweight

Flyweight
Compartilha o estado comum entre muitos objetos para suportar grandes quantidades deles com baixo custo de memória.

Quando você precisa criar milhões de objetos parecidos, a memória vira o gargalo. O Flyweight separa o estado intrínseco (compartilhável, imutável — ex.: a textura de uma árvore) do estado extrínseco (único de cada instância — ex.: a posição x,y) e reaproveita o primeiro entre todas as instâncias.

Solução (TypeScript)
flyweight.ts · typescript
// Estado intrínseco — pesado e compartilhável (textura, cor, sprite)
class TreeType {
  constructor(
    public readonly name: string,
    public readonly texture: string,
  ) {}
  render(x: number, y: number) {
    return `${this.name}@(${x},${y})`;
  }
}

// Fábrica que garante UMA instância por combinação intrínseca
class TreeTypeFactory {
  private static pool = new Map<string, TreeType>();
  static get(name: string, texture: string): TreeType {
    const key = `${name}:${texture}`;
    let type = this.pool.get(key);
    if (!type) this.pool.set(key, (type = new TreeType(name, texture)));
    return type; // reaproveitado entre milhares de árvores
  }
}

// Estado extrínseco (x, y) fica fora; o tipo é compartilhado
const forest = Array.from({ length: 100_000 }, (_, i) => ({
  x: i % 500, y: Math.floor(i / 500),
  type: TreeTypeFactory.get("Carvalho", "oak.png"), // sempre o mesmo objeto
}));
◤ Use quando
  • A aplicação cria um número enorme de objetos e a memória é um problema real.
  • Boa parte do estado desses objetos é idêntica e pode ser extraída e compartilhada.
  • O estado compartilhado é imutável.
◦ Evite quando
  • Você tem poucos objetos — a otimização não compensa a complexidade.
  • O estado é majoritariamente único por instância (pouco a compartilhar).

Proxy

Proxy
Fornece um substituto ou marcador para outro objeto, controlando o acesso a ele.

Um Proxy tem a mesma interface do objeto real e fica na frente dele, interceptando chamadas. Isso habilita carregamento preguiçoso (virtual proxy), cache, controle de acesso (protection proxy) ou logging — sem que o cliente perceba que não está falando com o objeto real.

Solução (TypeScript)
proxy.ts · typescript
interface ImageService {
  fetch(id: string): Promise<string>;
}

// Objeto real — operação cara (rede, disco)
class RemoteImageService implements ImageService {
  async fetch(id: string) {
    console.log(`baixando ${id}...`);
    return `bytes-de-${id}`;
  }
}

// Proxy de cache: mesma interface, controla o acesso ao real
class CachingImageProxy implements ImageService {
  private cache = new Map<string, string>();
  constructor(private real: ImageService) {}
  async fetch(id: string) {
    if (!this.cache.has(id)) {
      this.cache.set(id, await this.real.fetch(id));
    }
    return this.cache.get(id)!;     // 2ª chamada não bate na rede
  }
}

const images = new CachingImageProxy(new RemoteImageService());
await images.fetch("capa.png"); // baixa
await images.fetch("capa.png"); // cache
O JavaScript tem um Proxy nativo

A linguagem oferece o objeto Proxy com traps (get, set, has...) para interceptar operações de forma genérica — base de bibliotecas reativas como o Vue. O padrão GoF é mais explícito e tipado; o Proxy nativo é ideal para interceptação dinâmica e metaprogramação.

◤ Use quando
  • Você precisa de carregamento preguiçoso de um recurso caro (virtual proxy).
  • Quer adicionar cache, controle de acesso, logging ou rate-limiting transparentes.
  • Precisa representar localmente um objeto que vive em outro processo/servidor.
◦ Evite quando
  • Você não precisa de nenhum controle extra — o acesso direto é mais simples.
  • A indireção esconde latência ou efeitos colaterais importantes de quem chama.

Eles descrevem não só a estrutura, mas os padrões de comunicação: quem fala com quem, quando, e como o controle flui pelo sistema. Muitos deles aparecem o tempo todo em frameworks Node (Express, NestJS, RxJS).

Strategy

Strategy
Define uma família de algoritmos, encapsula cada um e os torna intercambiáveis em tempo de execução.

Em vez de embutir várias variações de um algoritmo com if/else, você extrai cada variação para um objeto com a mesma interface e injeta a estratégia desejada. É o gêmeo prático do princípio Aberto/Fechado: novas estratégias entram sem tocar no contexto.

Solução (TypeScript)
strategy.ts · typescript
// A família de algoritmos compartilha uma interface
interface ShippingStrategy {
  calculate(weightKg: number): number;
}

class StandardShipping implements ShippingStrategy {
  calculate(w: number) { return w * 2; }
}
class ExpressShipping implements ShippingStrategy {
  calculate(w: number) { return w * 2 + 15; }
}
class FreeShipping implements ShippingStrategy {
  calculate() { return 0; }
}

// O contexto recebe a estratégia e delega — não conhece os detalhes
class Cart {
  constructor(private shipping: ShippingStrategy) {}
  setShipping(s: ShippingStrategy) { this.shipping = s; }
  shippingCost(weightKg: number) { return this.shipping.calculate(weightKg); }
}

const cart = new Cart(new StandardShipping());
cart.shippingCost(3);            // 6
cart.setShipping(new ExpressShipping());
cart.shippingCost(3);            // 21
Em TypeScript, uma estratégia costuma ser só uma função

Quando a estratégia não tem estado, a interface vira um tipo de função: type ShippingStrategy = (w: number) => number. Aí você passa (w) => w * 2 direto. Use a versão com classes quando houver estado, configuração ou múltiplos métodos por estratégia.

◤ Use quando
  • Você tem várias maneiras de fazer a mesma coisa e quer trocar entre elas em runtime.
  • Quer eliminar condicionais grandes que selecionam comportamento.
  • Os algoritmos têm a mesma assinatura de entrada/saída.
◦ Evite quando
  • Existe só um algoritmo e nenhuma perspectiva de variação.
  • As variações são triviais — um simples parâmetro ou função inline resolve.

Observer

Observer
Define uma dependência um-para-muitos: quando um objeto muda de estado, todos os seus dependentes são notificados automaticamente.

O Subject mantém uma lista de observers e os avisa a cada mudança, sem saber quem são concretamente — só que implementam a interface de notificação. É a espinha dorsal de sistemas reativos e orientados a eventos. O EventEmitter do Node e o RxJS são Observer levado a sério.

Solução (TypeScript)
observer.ts · typescript
type Observer<T> = (data: T) => void;

class Subject<T> {
  private observers = new Set<Observer<T>>();

  subscribe(fn: Observer<T>): () => void {
    this.observers.add(fn);
    return () => this.observers.delete(fn); // unsubscribe
  }
  notify(data: T) {
    for (const fn of this.observers) fn(data);
  }
}

// Exemplo: preço de uma ação notificando vários painéis
const price = new Subject<number>();

const unsub = price.subscribe((p) => console.log(`Gráfico: ${p}`));
price.subscribe((p) => {
  if (p > 100) console.log(`Alerta: preço alto (${p})`);
});

price.notify(90);   // só o gráfico reage
price.notify(120);  // gráfico + alerta
unsub();            // o gráfico para de ouvir
◤ Use quando
  • Uma mudança em um objeto exige atualizar outros, e você não sabe quantos antecipadamente.
  • Quer baixo acoplamento entre quem emite e quem reage ao evento.
  • Está modelando eventos, notificações ou fluxos de dados reativos.
◦ Evite quando
  • Há um único dependente fixo — uma chamada direta é mais clara.
  • A cadeia de notificações em cascata fica difícil de depurar (cuidado com loops de eventos).

Command

Command
Encapsula uma requisição como um objeto, permitindo parametrizar, enfileirar, registrar e desfazer operações.

Transformar uma ação em objeto (com um método execute()) desacopla quem pede de quem executa. A partir daí você ganha superpoderes: filas de tarefas, histórico, log de operações e — o caso de uso estrela — desfazer/refazer via um método undo().

Solução (TypeScript)
command.ts · typescript
interface Command {
  execute(): void;
  undo(): void;
}

class TextDocument {
  content = "";
}

class AppendCommand implements Command {
  constructor(private doc: TextDocument, private text: string) {}
  execute() { this.doc.content += this.text; }
  undo()    { this.doc.content = this.doc.content.slice(0, -this.text.length); }
}

// Invoker: executa comandos e mantém o histórico para desfazer
class CommandHistory {
  private history: Command[] = [];
  run(cmd: Command) { cmd.execute(); this.history.push(cmd); }
  undo() { this.history.pop()?.undo(); }
}

const doc = new TextDocument();
const editor = new CommandHistory();
editor.run(new AppendCommand(doc, "Olá"));
editor.run(new AppendCommand(doc, ", mundo"));
console.log(doc.content); // "Olá, mundo"
editor.undo();
console.log(doc.content); // "Olá"
◤ Use quando
  • Você precisa de desfazer/refazer, filas de operações, agendamento ou log de ações.
  • Quer desacoplar o objeto que invoca a operação do que sabe executá-la.
  • Operações devem ser tratadas como dados (serializadas, transmitidas, repetidas).
◦ Evite quando
  • A ação é uma chamada direta simples, sem necessidade de histórico ou enfileiramento.
  • Envolver tudo em objetos-comando só adiciona cerimônia.

Chain of Responsibility

Chain of Responsibility
Passa uma requisição por uma cadeia de handlers; cada um decide se a trata ou a repassa ao próximo.

Você desacopla o remetente da requisição dos seus possíveis receptores ligando-os em uma cadeia. Cada elo trata o que é da sua alçada e delega o resto. É exatamente o modelo de middlewares do Express e dos guards/interceptors do NestJS.

Solução (TypeScript)
chain.ts · typescript
abstract class Handler {
  private next?: Handler;
  setNext(h: Handler): Handler { this.next = h; return h; } // encadeável
  handle(request: { role: string; amount: number }): string {
    return this.next ? this.next.handle(request) : "rejeitado";
  }
}

class TeamLead extends Handler {
  handle(req: { role: string; amount: number }) {
    if (req.amount <= 1000) return "aprovado pelo líder";
    return super.handle(req); // delega adiante
  }
}
class Manager extends Handler {
  handle(req: { role: string; amount: number }) {
    if (req.amount <= 10000) return "aprovado pelo gerente";
    return super.handle(req);
  }
}
class Director extends Handler {
  handle(req: { role: string; amount: number }) {
    return "aprovado pelo diretor";
  }
}

const chain = new TeamLead();
chain.setNext(new Manager()).setNext(new Director());
console.log(chain.handle({ role: "dev", amount: 500 }));    // líder
console.log(chain.handle({ role: "dev", amount: 8000 }));   // gerente
console.log(chain.handle({ role: "dev", amount: 50000 }));  // diretor
◤ Use quando
  • Mais de um objeto pode tratar a requisição e você não sabe qual de antemão.
  • Quer processar algo em etapas, cada uma podendo interromper ou seguir adiante.
  • Está construindo pipelines de middleware, validação ou aprovação.
◦ Evite quando
  • Sempre o mesmo objeto trata a requisição — a cadeia é desnecessária.
  • Nenhum handler tratar a requisição (cair no fim sem ninguém responder) é um risco silencioso.

Template Method

Template Method
Define o esqueleto de um algoritmo em um método, delegando etapas específicas para as subclasses.

A superclasse fixa a ordem dos passos em um método final, mas deixa um ou mais passos como métodos abstratos que as subclasses preenchem. Você reaproveita a estrutura e varia apenas os detalhes — o Hollywood Principle: "não nos chame, nós chamamos você".

Solução (TypeScript)
template-method.ts · typescript
abstract class DataExporter {
  // O template: ordem fixa, passos variáveis (não deve ser sobrescrito)
  export(rows: object[]): string {
    const header = this.formatHeader(rows);
    const body = rows.map((r) => this.formatRow(r)).join("");
    return this.wrap(header + body);
  }
  protected abstract formatHeader(rows: object[]): string;
  protected abstract formatRow(row: object): string;
  protected wrap(content: string): string { return content; } // hook opcional
}

class JsonExporter extends DataExporter {
  protected formatHeader() { return ""; }
  protected formatRow(row: object) { return JSON.stringify(row) + ","; }
  protected wrap(c: string) { return `[${c.replace(/,$/, "")}]`; }
}

class CsvExporter extends DataExporter {
  protected formatHeader(rows: object[]) {
    return Object.keys(rows[0] ?? {}).join(";") + "|";
  }
  protected formatRow(row: object) { return Object.values(row).join(";") + "|"; }
}

new JsonExporter().export([{ id: 1 }, { id: 2 }]); // [{"id":1},{"id":2}]
Template Method vs. Strategy

Os dois variam um algoritmo. O Template Method usa herança (a variação é fixada na subclasse, em tempo de compilação); o Strategy usa composição (a variação é um objeto injetado, trocável em runtime). Em código moderno, prefira composição quando puder escolher — mas o Template Method brilha quando há muita estrutura compartilhada e poucos pontos de variação.

◤ Use quando
  • Vários algoritmos têm a mesma estrutura geral e diferem só em alguns passos.
  • Você quer evitar duplicar o "esqueleto" em cada variação.
  • Faz sentido travar a ordem dos passos e abrir apenas pontos específicos.
◦ Evite quando
  • As variações não compartilham estrutura — herança seria forçada.
  • Você precisa trocar o comportamento em tempo de execução (use Strategy).

State

State
Permite que um objeto altere seu comportamento quando seu estado interno muda — como se trocasse de classe.

Em vez de um emaranhado de if (status === ...) espalhado por todos os métodos, cada estado vira um objeto que sabe como se comportar e para qual estado transicionar. O contexto delega ao objeto-estado atual. É uma máquina de estados explícita e extensível.

Solução (TypeScript)
state.ts · typescript
interface OrderState {
  next(order: Order): void;
  readonly label: string;
}

class PendingState implements OrderState {
  label = "pendente";
  next(order: Order) { order.setState(new PaidState()); }
}
class PaidState implements OrderState {
  label = "pago";
  next(order: Order) { order.setState(new ShippedState()); }
}
class ShippedState implements OrderState {
  label = "enviado";
  next() { /* estado final: nada a fazer */ }
}

class Order {
  private state: OrderState = new PendingState();
  setState(s: OrderState) { this.state = s; }
  advance() { this.state.next(this); }
  get status() { return this.state.label; }
}

const order = new Order();
order.advance(); console.log(order.status); // pago
order.advance(); console.log(order.status); // enviado
Em TypeScript, considere uma union discriminada

Para máquinas de estado simples, um tipo type State = { kind: "pending" } | { kind: "paid" } | ... com um switch no kind dá segurança de tipos (checagem de exaustividade pelo compilador) sem uma classe por estado. Use o padrão clássico quando cada estado tem comportamento rico e muitos métodos.

◤ Use quando
  • O comportamento do objeto depende fortemente do seu estado e muda em runtime.
  • Você tem condicionais grandes e repetidas baseadas no estado atual.
  • As transições entre estados são bem definidas e você quer torná-las explícitas.
◦ Evite quando
  • Há poucos estados e pouca lógica — um enum com um switch resolve.
  • Os estados quase não diferem em comportamento.

Iterator

Iterator
Fornece uma forma de acessar sequencialmente os elementos de uma coleção sem expor sua representação interna.

O Iterator separa percorrer de armazenar: o cliente caminha pela coleção sem saber se por baixo há um array, uma árvore ou um grafo. O JavaScript embute esse padrão no protocolo de iteração — qualquer objeto com [Symbol.iterator] funciona em for...of, spread e desestruturação.

Solução (TypeScript)
iterator.ts · typescript
// Coleção custom que adere ao protocolo de iteração do JS
class NumberRange implements Iterable<number> {
  constructor(private start: number, private end: number, private step = 1) {}

  [Symbol.iterator](): Iterator<number> {
    let current = this.start;
    const { end, step } = this;
    return {
      next(): IteratorResult<number> {
        if (current <= end) {
          const value = current;
          current += step;
          return { value, done: false };
        }
        return { value: undefined as never, done: true };
      },
    };
  }
}

// Funciona com toda a maquinaria nativa da linguagem
const range = new NumberRange(0, 10, 2);
for (const n of range) console.log(n);  // 0 2 4 6 8 10
console.log([...range]);                // [0, 2, 4, 6, 8, 10]

// Geradores são a forma idiomática de criar iteradores em JS
function* fibonacci(limit: number) {
  let [a, b] = [0, 1];
  while (a <= limit) { yield a; [a, b] = [b, a + b]; }
}
console.log([...fibonacci(20)]); // [0, 1, 1, 2, 3, 5, 8, 13]
◤ Use quando
  • Você quer percorrer uma estrutura complexa sem revelar como ela é armazenada.
  • Precisa de múltiplas formas de iteração (ordem, filtro) sobre a mesma coleção.
  • Quer integrar sua coleção com for...of, spread e desestruturação.
◦ Evite quando
  • Você está sobre um array/Map comum — os iteradores nativos já bastam.
  • Reimplementar o protocolo manualmente onde um gerador resolveria é trabalho à toa.

Mediator

Mediator
Centraliza a comunicação entre objetos em um mediador, reduzindo as dependências diretas entre eles.

Quando muitos objetos se comunicam diretamente, você obtém um grafo "todos com todos" difícil de manter. O Mediator coloca um intermediário no centro: cada objeto fala só com o mediador, que coordena os demais. As peças passam de fortemente acopladas a independentes.

Solução (TypeScript)
mediator.ts · typescript
interface ChatMediator {
  send(from: string, message: string): void;
  register(user: User): void;
}

class User {
  constructor(public name: string, private mediator: ChatMediator) {
    mediator.register(this);
  }
  send(message: string) { this.mediator.send(this.name, message); }
  receive(from: string, message: string) {
    console.log(`[${this.name}] recebeu de ${from}: ${message}`);
  }
}

// O mediador concreto conhece os participantes e orquestra as trocas
class ChatRoom implements ChatMediator {
  private users = new Map<string, User>();
  register(user: User) { this.users.set(user.name, user); }
  send(from: string, message: string) {
    for (const [name, user] of this.users) {
      if (name !== from) user.receive(from, message); // broadcast
    }
  }
}

const room = new ChatRoom();
const ana = new User("Ana", room);
const beto = new User("Beto", room);
ana.send("olá a todos");   // Beto recebe; usuários não se conhecem entre si
◤ Use quando
  • Um conjunto de objetos se comunica de formas complexas e interdependentes.
  • Você quer reaproveitar componentes sem arrastar junto todas as suas conexões.
  • A lógica de coordenação merece viver em um lugar só (ex.: orquestração de um formulário/UI).
◦ Evite quando
  • Há poucos objetos com comunicação simples — o mediador vira intermediário inútil.
  • O mediador começa a inchar e virar um "objeto-deus" que sabe demais.

Memento

Memento
Captura e externaliza o estado interno de um objeto, sem violar seu encapsulamento, para poder restaurá-lo depois.

O Memento é a maquinaria por trás do undo e dos snapshots. O objeto (originator) produz um memento opaco com seu estado; um cuidador (caretaker) guarda esses mementos sem inspecioná-los; depois, o originator usa um memento para voltar a um estado anterior. Combina muito bem com Command.

Solução (TypeScript)
memento.ts · typescript
// Memento: instantâneo imutável e opaco do estado
class EditorMemento {
  constructor(public readonly content: string) {}
}

// Originator: cria e restaura a partir de mementos
class Editor {
  private content = "";
  type(text: string) { this.content += text; }
  getContent() { return this.content; }
  save(): EditorMemento { return new EditorMemento(this.content); }
  restore(memento: EditorMemento) { this.content = memento.content; }
}

// Caretaker: guarda o histórico sem olhar dentro do memento
class History {
  private snapshots: EditorMemento[] = [];
  push(m: EditorMemento) { this.snapshots.push(m); }
  pop(): EditorMemento | undefined { return this.snapshots.pop(); }
}

const editor = new Editor();
const history = new History();

editor.type("primeira linha");
history.push(editor.save());        // checkpoint
editor.type(" + rascunho");
editor.restore(history.pop()!);     // desfaz
console.log(editor.getContent());   // "primeira linha"
◤ Use quando
  • Você precisa de undo/redo, checkpoints ou snapshots do estado de um objeto.
  • Quer salvar/restaurar estado sem expor os campos internos do objeto.
  • Está implementando histórico, versionamento ou "viagem no tempo" de estado.
◦ Evite quando
  • O estado é trivial de recriar — salvar/restaurar é exagero.
  • Os snapshots são grandes e frequentes a ponto de pesar na memória (considere deltas).

Padrão Comportamental

Visitor

Visitor
Separa um algoritmo da estrutura de objetos sobre a qual ele opera, permitindo adicionar novas operações sem alterar as classes.

O Visitor resolve uma tensão: você quer adicionar operações novas a uma hierarquia de objetos (formas, nós de uma AST) sem editar cada classe a cada operação. Cada elemento "aceita" um visitante e chama o método certo dele (double dispatch). Novas operações = novos visitantes; as classes da estrutura ficam intactas.

Solução (TypeScript)
visitor.ts · typescript
interface ShapeVisitor<R> {
  visitCircle(c: Circle): R;
  visitSquare(s: Square): R;
}
interface Shape {
  accept<R>(visitor: ShapeVisitor<R>): R;
}

class Circle implements Shape {
  constructor(public radius: number) {}
  accept<R>(v: ShapeVisitor<R>) { return v.visitCircle(this); }
}
class Square implements Shape {
  constructor(public side: number) {}
  accept<R>(v: ShapeVisitor<R>) { return v.visitSquare(this); }
}

// Nova operação = novo visitante, sem tocar em Circle/Square
class AreaVisitor implements ShapeVisitor<number> {
  visitCircle(c: Circle) { return Math.PI * c.radius ** 2; }
  visitSquare(s: Square) { return s.side ** 2; }
}

const shapes: Shape[] = [new Circle(2), new Square(3)];
const area = new AreaVisitor();
console.log(shapes.map((s) => s.accept(area))); // [12.56..., 9]
Em TypeScript, muitas vezes a union discriminada vence

O Visitor existe para contornar a falta de pattern matching em linguagens OO clássicas. Em TypeScript, um tipo type Shape = Circle | Square com switch (shape.kind) e checagem de exaustividade entrega o mesmo resultado com muito menos cerimônia. Reserve o Visitor para hierarquias estáveis em que você adiciona operações o tempo todo, mas raramente adiciona tipos novos.

◤ Use quando
  • Você precisa adicionar muitas operações distintas a uma hierarquia estável de classes.
  • As operações não pertencem naturalmente às classes (relatórios, exportações, análises).
  • A estrutura (tipos de nós) muda pouco; as operações mudam muito.
◦ Evite quando
  • Você adiciona tipos novos com frequência — cada novo tipo obriga a tocar todos os visitantes.
  • Em TypeScript: uma union discriminada com switch resolve de forma mais simples.

Padrão Comportamental

Interpreter

Interpreter
Dada uma linguagem, define uma representação para sua gramática e um interpretador que avalia sentenças dessa linguagem.

O menos usado do catálogo, e o mais especializado. Cada regra da gramática vira uma classe; uma sentença vira uma árvore dessas classes que sabe se avaliar. É útil para mini-linguagens: filtros, regras de negócio configuráveis, expressões booleanas. Para linguagens reais, use um parser dedicado — mas a ideia é a base de muitos motores de regras.

Solução (TypeScript)
interpreter.ts · typescript
type Context = Record<string, boolean>;

interface Expression {
  interpret(context: Context): boolean;
}

// Expressão terminal: consulta uma variável no contexto
class Variable implements Expression {
  constructor(private name: string) {}
  interpret(ctx: Context) { return ctx[this.name] ?? false; }
}

// Expressões não-terminais: combinam outras expressões
class And implements Expression {
  constructor(private left: Expression, private right: Expression) {}
  interpret(ctx: Context) {
    return this.left.interpret(ctx) && this.right.interpret(ctx);
  }
}
class Or implements Expression {
  constructor(private left: Expression, private right: Expression) {}
  interpret(ctx: Context) {
    return this.left.interpret(ctx) || this.right.interpret(ctx);
  }
}

// Monta a árvore: (isAdmin OR isOwner) AND isActive
const rule = new And(
  new Or(new Variable("isAdmin"), new Variable("isOwner")),
  new Variable("isActive"),
);
console.log(rule.interpret({ isOwner: true, isActive: true }));  // true
console.log(rule.interpret({ isAdmin: true, isActive: false })); // false
◤ Use quando
  • Você tem uma mini-linguagem ou gramática simples e estável para avaliar.
  • As sentenças podem ser representadas como árvores de sintaxe abstrata.
  • Está construindo um motor de regras, filtros ou expressões configuráveis.
◦ Evite quando
  • A gramática é complexa ou muda muito — um parser/biblioteca dedicada é melhor.
  • Você só precisa avaliar uma expressão fixa — não há linguagem a interpretar.

Você talvez não precise (de um padrão)

Os padrões da Gang of Four foram catalogados em 1994, pensando em linguagens como C++ e Smalltalk — onde funções não são valores de primeira classe e o sistema de tipos é menos expressivo. TypeScript mudou o jogo. Vários padrões clássicos resolvem, na prática, limitações que a linguagem já não tem. Saber quando não aplicar um padrão é tão sênior quanto saber aplicá-lo.

Isto não é "padrões são ruins". É reconhecer que metade do catálogo descreve coisas que, em TypeScript idiomático, você expressa com **funções, objetos literais e *union types*** — com menos código e mais clareza. Vejamos os casos mais comuns.

Funções de primeira classe → Strategy e Command simples

Quando uma estratégia ou comando não tem estado, a hierarquia de classes vira ruído. Uma função (ou um mapa de funções) entrega a mesma intercambialidade:

strategy-as-function.ts · typescript
// "Strategy" clássico: interface + uma classe por algoritmo
interface Discount { apply(total: number): number; }
class HalfOff implements Discount { apply(t: number) { return t * 0.5; } }
class NoDiscount implements Discount { apply(t: number) { return t; } }

// TypeScript idiomático: o tipo É uma função; as estratégias são valores
type Discount = (total: number) => number;

const discounts: Record<string, Discount> = {
  black_friday: (t) => t * 0.5,
  none:         (t) => t,
};

discounts.black_friday(100); // 50 — sem new, sem classes

Union types discriminadas → State e Visitor

Boa parte do State e do Visitor existe para simular pattern matching. TypeScript tem isso de fábrica com uniões discriminadas e checagem de exaustividade — o compilador te obriga a tratar todos os casos, e cada caso só enxerga os campos que fazem sentido:

state-as-union.ts · typescript
// Cada estado carrega exatamente os dados que lhe pertencem
type RequestState =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: string[] }
  | { status: "error"; message: string };

function render(state: RequestState): string {
  switch (state.status) {
    case "idle":    return "aguardando";
    case "loading": return "carregando...";
    case "success": return state.data.join(", "); // data só existe aqui
    case "error":   return state.message;          // message só existe aqui
    // sem default: se um novo status surgir, o compilador acusa o caso faltante
  }
}

Objetos literais e módulos → Singleton e Factory triviais

Um módulo ES já é uma instância única e compartilhada — export const config = {...} é um singleton sem a classe, sem o getInstance() e sem o estado global escondido. E uma fábrica simples muitas vezes é apenas uma função que retorna o objeto certo, não uma hierarquia de creators.

Tipos como contratos → "parse, don’t validate"

Em vez de validar o mesmo dado repetidamente Brasil afora, transforme-o uma vez em um tipo que prova que a validação aconteceu. Estados inválidos deixam de ser representáveis — e o compilador passa a carregar a garantia para você:

parse-dont-validate.ts · typescript
// Um "branded type": um string que o sistema de tipos sabe ser um e-mail válido
type Email = string & { readonly __brand: "Email" };

function parseEmail(input: string): Email | null {
  return /^[^@]+@[^@]+\.[^@]+$/.test(input) ? (input as Email) : null;
}

function sendWelcome(to: Email) { /* ... */ }

const maybe = parseEmail(userInput);
if (maybe) sendWelcome(maybe); // a partir daqui, "Email" garante a validação
O verdadeiro anti-pattern: abstração prematura

Aplicar um padrão "porque é boa prática", antes de o problema existir, é over-engineering — você paga o custo da indireção sem colher benefício. Duas heurísticas valiosas: a regra de três (só extraia uma abstração no terceiro caso de duplicação real, não no primeiro) e o YAGNI (You Aren’t Gonna Need It — não construa flexibilidade especulativa). Um pouco de duplicação é mais barato do que a abstração errada. Padrões são uma resposta a uma dor sentida, não um requisito de entrada.

A conclusão para o restante deste e-book: aprenda os 23 padrões a fundo — eles afiam seu vocabulário e sua visão de design. Mas, ao codar em TypeScript, sempre pergunte antes: "a linguagem já não resolve isto?" Quando a resposta for sim, o código mais simples vence. Voltamos a esse tema, com as mãos no código, no Dia 29.

Quebrando a classe que faz tudo

InicianteSRPRefatoração

Toda base de código tem uma. A classe que nasceu pequena e hoje lê dados, formata relatório, salva no banco e ainda dispara e-mail — uma mudança em qualquer ponto arrisca quebrar os outros três.

O desafio

Você recebeu o ReportService abaixo. Ele viola o SRP acumulando três responsabilidades. Refatore-o em unidades coesas, cada uma com uma única razão para mudar, e deixe uma peça orquestradora apenas conectando-as.

dia-01.ts · typescript
// Ponto de partida — três responsabilidades em uma classe
class ReportService {
  generate(userId: string): void {
    // 1) acesso a dados
    const data = { name: "Ana", sales: 4200 }; // imagine uma query
    // 2) formatação
    const html = `<h1>${data.name}</h1><p>Vendas: ${data.sales}</p>`;
    // 3) entrega
    console.log("enviando e-mail com:", html);
  }
}
Requisitos
  • Identifique as responsabilidades distintas: fonte de dados, formatação e entrega.
  • Crie uma classe (ou função) por responsabilidade — ReportRepository, ReportFormatter, ReportSender.
  • Mantenha regra de negócio separada de I/O (rede, disco, console).
  • Componha tudo em um orquestrador fino que não contém a lógica das partes.
✓ Definição de pronto
  • Trocar o meio de entrega (e-mail &rarr; Slack) não exige tocar na formatação nem no acesso a dados.
  • Cada classe tem um nome que descreve uma tarefa e cabe em poucas linhas.

Aberto para o futuro

InicianteOCPStrategy

Um switch que cresce a cada novo caso é o cheiro clássico de violação do princípio Aberto/Fechado: cada novidade obriga a reabrir — e arriscar — um código que já funcionava.

O desafio

Dado um cálculo de frete que usa um switch por transportadora, refatore para que adicionar uma nova transportadora não exija editar a classe calculadora.

Requisitos

  • Cada transportadora vira uma implementação isolada da interface.
  • Use um registro (um Map de chave &rarr; método) para resolver a transportadora.

Quando herdar quebra

IntermediárioLSPModelagem

O exemplo canônico: Square extends Rectangle. Parece correto na geometria, mas viola o contrato — mudar a largura de um quadrado muda a altura, e quem espera um retângulo se surpreende.

O desafio

Modele formas com area() de modo que toda substituição respeite o contrato da base. Encontre o ponto onde a herança força um comportamento que viola Liskov e remodele a hierarquia.

dia-03.ts · typescript
// Ponto de partida problemático
class Rectangle {
  constructor(protected w: number, protected h: number) {}
  setWidth(w: number)  { this.w = w; }
  setHeight(h: number) { this.h = h; }
  area() { return this.w * this.h; }
}
// Square "é um" Rectangle? setWidth quebra a expectativa de quem usa Rectangle.
class Square extends Rectangle {
  setWidth(w: number)  { this.w = w; this.h = w; }  // efeito colateral surpresa
  setHeight(h: number) { this.w = h; this.h = h; }
}
Requisitos
  • Escreva uma função growWidth(r: Rectangle) que falha (logicamente) ao receber um Square — para evidenciar a violação.
  • Remodele: prefira uma abstração comum (interface Shape { area(): number }) a uma herança forçada.
  • Garanta que nenhum subtipo lance "não suportado" nem ignore parâmetros do contrato.
✓ Definição de pronto
  • Qualquer Shape pode substituir outro em código que só depende de area(), sem surpresas.
  • Não há método sobrescrito que enrijeça pré/pós-condições da base.

Interfaces enxutas

InicianteISPCoesão

Forçar uma classe a implementar métodos que não usa gera implementações vazias ou que lançam exceções — sinal de uma interface "gorda" demais.

O desafio

Você tem uma interface Worker com work(), eat() e sleep(). Um RobotWorker não come nem dorme. Segregue a interface em contratos pequenos e focados, e faça cada classe implementar só o que lhe cabe.

Requisitos

  • HumanWorker implementa as três; RobotWorker implementa apenas Workable.
  • Nenhuma classe deve ter método vazio ou que lança Error("não suportado").
  • Funções clientes devem depender da menor interface de que precisam.
✓ Definição de pronto
  • Nenhum implementador é obrigado a depender de um método que não usa.
  • Uma função que só precisa de work() aceita tanto humano quanto robô, tipada por Workable.

Dependa de abstrações

IntermediárioDIPInjeção de Dependência

Quando uma classe de alto nível dá new em uma classe concreta de baixo nível, ela fica casada com um detalhe. O DIP inverte a seta: ambos passam a depender de uma abstração.

O desafio

Um NotificationService instancia diretamente um SmtpEmailClient. Inverta a dependência: o serviço deve depender de uma interface, e a implementação concreta deve ser injetada de fora.

dia-05.ts · typescript
// Ponto de partida — acoplado ao detalhe concreto
class SmtpEmailClient {
  send(to: string, body: string) { /* SMTP... */ }
}
class NotificationService {
  private client = new SmtpEmailClient(); // <-- dependência concreta "soldada"
  notify(user: string, msg: string) { this.client.send(user, msg); }
}
Requisitos
  • Defina interface MessageClient { send(to: string, body: string): void }.
  • NotificationService recebe um MessageClient pelo construtor (injeção).
  • Implemente ao menos duas concretas: SmtpEmailClient e FakeClient (para testes).
  • A montagem (qual implementação usar) acontece em um ponto de composição, fora do serviço.
✓ Definição de pronto
  • Trocar SMTP por SMS/Slack não muda uma linha do NotificationService.
  • Você testa o serviço com o FakeClient, sem rede, verificando o que foi enviado.

A fábrica que decide

IntermediárioFactory MethodConfiguração

Construindo passo a passo

IntermediárioBuilderAPI fluente

Construtores com oito parâmetros, metade opcionais, viram um campo minado de argumentos posicionais. O Builder troca isso por uma API que se lê quase como uma frase.

O desafio

Implemente um QueryBuilder fluente para montar uma consulta SQL simples (SELECT ... FROM ... WHERE ... LIMIT ...), com validação no build(). Cada etapa retorna o próprio builder para encadear.

class QueryBuilder {

  where(condition: string): this  { /* ... */ return this; }

Construa um interpretador para expressões booleanas com variáveis (AND, OR, NOT, variável). Monte uma árvore como (isAdmin OR isOwner) AND isActive e avalie-a contra um contexto de valores.

Requisitos
  • Defina interface Expression { interpret(ctx): boolean }.
  • Implemente a expressão terminal (variável) e as não-terminais (And, Or, Not).
  • Monte uma árvore aninhada e avalie-a com diferentes contextos.
  • A avaliação deve ser puramente recursiva, sem condicionais sobre "tipo de nó".
✓ Definição de pronto
  • A mesma árvore retorna resultados corretos para contextos diferentes.
  • Adicionar um operador novo (ex.: Xor) é só uma classe a mais.
Onde isto aparece de verdade

Motores de regras de negócio, filtros configuráveis por usuário e linguagens de query simples usam exatamente esta ideia. Para linguagens reais, porém, use um parser dedicado — o Interpreter brilha em gramáticas pequenas e estáveis.

Removendo o que sobra

AvançadoRefatoraçãoYAGNITypeScript idiomático

Saber aplicar padrões é metade do ofício sênior; a outra metade é saber remover os que não pagam o próprio custo. Hoje você inverte o sentido da Parte I: pega código sobre-projetado e o simplifica usando os recursos da linguagem.

O desafio

Você recebeu o módulo abaixo, escrito por alguém apaixonado por padrões. Ele tem uma fábrica, um singleton e uma hierarquia de estratégias — tudo para algo trivial. Refatore para TypeScript idiomático, mantendo o comportamento e cortando a cerimônia.

dia-29-antes.ts · typescript
// Código sobre-projetado: muito padrão para pouca necessidade
interface GreetingStrategy { greet(name: string): string; }
class FormalGreeting implements GreetingStrategy { greet(n: string) { return `Prezado ${n}`; } }
class CasualGreeting implements GreetingStrategy { greet(n: string) { return `E aí, ${n}`; } }

class GreetingStrategyFactory {
  static create(kind: string): GreetingStrategy {
    if (kind === "formal") return new FormalGreeting();
    return new CasualGreeting();
  }
}

class GreeterSingleton {
  private static instance: GreeterSingleton;
  private constructor() {}
  static getInstance() { return (GreeterSingleton.instance ??= new GreeterSingleton()); }
  greet(kind: string, name: string) {
    return GreetingStrategyFactory.create(kind).greet(name);
  }
}
// Uso: GreeterSingleton.getInstance().greet("formal", "Ana")
Requisitos
  • Substitua a hierarquia de estratégias por um Record de funções (estratégia sem estado &rarr; função).
  • Elimine o Singleton: uma função exportada do módulo basta — sem estado global.
  • Dispense a fábrica: a seleção vira um acesso a mapa, não uma classe.
  • Preserve o comportamento observável e a segurança de tipos (considere um tipo literal para kind).
✓ Definição de pronto
  • O mesmo uso (greet("formal", "Ana")) continua funcionando, com uma fração do código.
  • Você consegue justificar, caso a caso, por que cada padrão removido era desnecessário aqui — e em que cenário ele voltaria a fazer sentido.
O critério

A pergunta-guia não é "isto é um padrão?", e sim "qual dor real este padrão alivia neste código?". Sem dor, é peso morto. Padrão é resposta a um problema sentido — nunca um requisito de entrada. Releia a seção Você talvez não precise enquanto refatora.

Projeto final: o motor de pedidos

AvançadoCapstoneArquiteturaMulti-pattern

O dia em que tudo se junta. Você vai construir um pequeno — porém real — motor de processamento de pedidos em Node + TypeScript, combinando vários padrões onde cada um resolve uma dor concreta. O objetivo não é usar muitos padrões: é usar o certo em cada ponto.

O projeto

Um serviço que recebe um pedido, valida, calcula preço final, processa o pagamento, avança pelo ciclo de vida e notifica os interessados — com trilha de auditoria. Monte-o compondo os padrões da Parte I:

Arquitetura sugerida
  • Chain of Responsibility — pipeline de validação do pedido (itens &rarr; estoque &rarr; endereço).
  • Strategy — cálculo de frete e de desconto intercambiáveis.
  • Decorator (ou composição de funções) — modificadores de preço empilháveis (cupom, taxa de região).
  • Factory Method — criação do gateway de pagamento conforme o método escolhido.
  • State — ciclo de vida do pedido (pendente &rarr; pago &rarr; enviado), com transições válidas.
  • Observer — ao mudar de estado, notifica e-mail, estoque e métricas, desacoplados.
  • Command — cada operação relevante vira um comando registrado para auditoria (e undo onde fizer sentido).
dia-30.ts · typescript
// Esqueleto de composição — o "ponto de montagem" do sistema
class OrderEngine {
  constructor(
    private validators: Handler,            // Chain of Responsibility
    private shipping: ShippingStrategy,     // Strategy
    private gatewayFactory: PaymentFactory, // Factory Method
    private events: Subject<OrderEvent>,    // Observer
  ) {}

  process(order: Order, method: string): void {
    this.validators.handle(order);                 // valida ou rejeita
    const total = this.shipping.calculate(order);  // preço final
    this.gatewayFactory.create(method).pay(total); // pagamento
    order.advance();                               // State: -> pago
    this.events.notify({ type: "paid", order });   // notifica interessados
  }
}

Requisitos
  • Cada padrão deve resolver uma dor real do fluxo — nada de padrão decorativo.
  • As peças se conectam por abstrações (interfaces), não por classes concretas (DIP).
  • O fluxo feliz e ao menos um caminho de erro (validação falha, pagamento recusado) estão cobertos por testes.
  • Um README curto explica qual padrão você usou em cada ponto e por quê.
✓ Definição de pronto
  • Processar um pedido válido percorre validação &rarr; preço &rarr; pagamento &rarr; estado &rarr; notificação.
  • Trocar uma estratégia de frete, um gateway ou um observador não quebra o resto do motor.
  • Você é capaz de defender cada escolha de design — e apontar onde deliberadamente não usou um padrão.
